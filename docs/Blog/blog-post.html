When to Code From Scratch And When Not Reinvent The Wheel

Published 3 months ago5 min read

Early one in my coding-learning journey, I wanted every single piece of code to be original. I would read, view tutorials, and ask questions to developers, but I wouldn't even copy/paste code online. In retrospect, I did help a bit, but it was gruesomely slow and honestly quite boring. Now, I'm not knocking hand coding, but the more I kept "researching" online, I noticed that there were plenty of other people that were also stuck in the same issues that I was having. Not only have I not used this wonderful resource, but I had deprived myself from the learning experiences from other developers.

Inspiration From Other People's Frustration

There are too many resources available to list, and to see the software enginnering community help each other. From noobs to Big Four senior software engineers, people help each other with (mostly) well defined solutions to questions posted. You can definitely sharpen your "How To Ask Better Questions" approach when searching/filtering through StackOverflow's more than 16 million questions. The more straight approach would be to simply google your question (add "Solved" to your search). To my delight, researching online o whether professional developers use google to get help in their daily operations at work, let to a resounding YES!

Top Five Websites For Programming Questions

There are plenty of resources to see similiar issues from other developers or to post a question and hopefully obtain an answer.

Here are some of my favorites:
  1. Stack Overflow
  2. Google Search
  3. Reddit r/Programming
  4. freeCodeCamp Forum
  5. CodeProject

But once again, this can feed the infamous Imposter Syndrome. Thankfully, we can rest assure that most developers go through this phase, and it is perfectly normal to find an ethical and comfortable balance of inspiration and hand coding that suits our needs.

I now look at these types of situations similiar to that of a using a cooking recipe for personal use. I can take the advice and descriptive solutions (prep work / cooking instructions), and I can implement and test the code (cook / bake), and only enjoy it for personal use (eat it).

Copyright law does not protect lists, or in our case the code example (treated as a list of ingredients), which means we can whip up a personal batch for our own use. It is important to note the main issue is knowing what a developer is allowed to do with code that is licensed, especially if you're planning on using the code to commercially.

My Conclusion

We can classify this use of code as borrowing, but it can also appear as stealing. This topic has been long discussed: Should Code/Software Be Considered Free?

This topic deserves its own series of book, so we won't dive too deep here. We will, however, acknowledge that there are hard cases (copyright laws and official licenses) that inherently surpasses any philosophy or principles. We will also acknowledge that the history of software, computer systems, and technology has been particularly interesting with its origin and in how this industry came to be. The characters from this history, for me, have shown personality traits and made belief-based decisions that do align with my way of thinking. This is one of my favorite things about programming, and the people who paved the way. From Grace Hopper to Linus Torvalds, to Guido van Rossum, to even the controversial Richard Stallman, this relatively young science has never (in my humble opinion) ever been neither dull nor stagnent.
